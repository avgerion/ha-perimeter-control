# Copilot Instructions: Perimeter Control System

## Core Architecture (CRITICAL)

This system uses a **two-device architecture**:

1. **Home Assistant Server** - Runs the custom integration (root level HA integration files)
2. **Raspberry Pi Target Device** - Remote Pi where services are deployed (e.g., `192.168.50.47`)

**NEVER confuse these two devices**. When referring to "the Pi" or "192.168.50.47", this is the **target device**, NOT the Home Assistant server.

## Communication Flow

```
Home Assistant ──SSH Deploy──► Raspberry Pi Target
Home Assistant ──HTTP API────► Raspberry Pi Target (port 8080)
Home Assistant ◄─WebSocket──── Raspberry Pi Target (real-time events)
```

## Directory Structure & Device Mapping

**Runs on Home Assistant:**
- `*.py, manifest.json` - Integration code (root level)
- `deployer.py` - SSH deployment logic  
- `coordinator.py` - API client for Pi communication
- `ha-integration/` - Deployment scripts and examples

**Deployed to Pi Target:**
- `remote_services/supervisor/` - Supervisor API (port 8080)
- `remote_services/dashboard_web/` - Dashboard web UI (port 3000)
- `system_services/` - Service descriptors and configs

## Key Deployment Process

1. Home Assistant integration uses SSH to connect to Pi target
2. HA deploys supervisor and services code to Pi via SFTP
3. HA starts systemd services on Pi
4. HA establishes ongoing API communication with Pi supervisor (port 8080)

## Common Misconceptions to Avoid

❌ **WRONG**: "The Pi runs Home Assistant"
✅ **CORRECT**: "The Pi runs the supervisor API that Home Assistant communicates with"

❌ **WRONG**: "Deploy to Home Assistant"  
✅ **CORRECT**: "Home Assistant deploys to the Pi target device"

❌ **WRONG**: "192.168.50.47 is the Home Assistant IP"
✅ **CORRECT**: "192.168.50.47 is the Pi target device IP"

## Development Guidelines

- When fixing deployment issues, focus on SSH/SFTP operations from HA → Pi
- When fixing API issues, focus on HTTP calls from HA → Pi (port 8080)
- When working with entities, remember they represent Pi device states queried via API
- All service runtime code runs on Pi, not HA
- Configuration files are deployed to Pi and managed remotely

## File Organization Logic

- **custom_components/**: Lives on HA server, handles UI and orchestration
- **remote_services/**: Gets deployed to Pi, contains actual service implementations  
- **system_services/**: Service descriptors and configs deployed to Pi
- **scripts/**: Development tools that can run from either device depending on purpose

## Testing Architecture

- Integration tests: Mock Pi API responses to test HA integration behavior
- Service tests: Run on Pi or Pi simulation to test supervisor and service logic
- Deployment tests: Use test Pi or container to validate SSH deployment process
